## 0. 一句话总论

STF3-New 是一个**需要训练的三分支 AI 生成视频检测模型**。它不是纯 training-free 方法：DINOv2/CLIP/XCLIP 等 foundation encoder 默认冻结，真正训练的是 spatial/temporal/frequency adapters、fusion transformer、主分类头和 auxiliary branch heads。当前最强主单模型权重是：

```text
runs/ood_stf3_new_224_fakew12/best.pt
```

对应配置：

```text
R7_224_FAKEW12 = stf3_new + image_size=224 + num_frames=8 + fake_loss_weight=1.2
```

在 OOD test 上使用验证集校准阈值 `0.0545` 后，主结果为：ACC=0.9430，F1=0.9499，Precision=0.9740，Recall=0.9271，FN=153，FP=52。

更完整地说，这个项目的核心论点可以概括为：

> 对 AI 生成视频检测而言，单一空间纹理、单一时序动态或单一频域伪影都不足以稳定覆盖未知生成器。STF3-New 将三类证据拆成可解释分支，并用 branch-token Transformer 做样本级融合；在 GenVideo-Val OOD split 上，最终三分支模型在相同训练与阈值校准协议下比 S+T 双分支模型减少 44 个 fake false negatives，说明 frequency evidence 与三分支融合对跨生成器检测有实证贡献。

本文档的阅读路线如下：

1. **先看模型**：理解 S/T/F 三个分支分别看什么、为什么这样设计、关键代码在哪里。
2. **再看基础实验**：Random 证明模型能学会任务，OOD 证明模型能否跨生成器泛化。
3. **重点看 OOD 优化**：这是本项目最有研究味道的部分，展示从问题诊断到最终主模型的完整心路。
4. **最后看 comparison**：它不是为了宣称全面超过所有外部方法，而是为了说明 STF3 的学术定位和边界。

# 第一部分：模型方法

## 1. 总体架构

STF3 的含义是 Spatial-Temporal-Frequency 三类证据融合。输入为均匀采样的视频帧：

```text
frames: [B, T, 3, H, W], value range [0, 1]
```

模型输出二分类 logits，经 softmax 得到 `fake_prob`：

```text
fake_prob = P(fake | video)
pred = 1 if fake_prob >= threshold else 0
```

总体结构为：

```text
Input video frames
  ├─ Spatial branch: frozen DINOv2/CLIP/XCLIP + temporal attention pooling
  ├─ Temporal branch: D3 second-order dynamics + ReStraV trajectory geometry
  └─ Frequency branch: Haar wavelet energy + radial FFT statistics
          ↓
  s_token, t_token, f_token
          ↓
  Branch-token Transformer / Gated concat
          ↓
  real/fake logits
```

架构图如下：

![STF3-New 架构图](docs/figures/STF3-New架构图.png)

### 1.1 任务形式化

本项目处理的是视频级二分类任务。给定一个视频 `V`，数据加载器从视频中采样 `T` 帧，得到：

```text
X = {x_1, x_2, ..., x_T}, x_t ∈ R^{3×H×W}
y ∈ {0, 1}, 0 = real, 1 = fake / AI-generated
```

STF3-New 学习一个函数：

```text
f_STF3(X) -> logits ∈ R^2
softmax(logits)[1] = fake_prob
```

最后的判断不是固定只能用 0.5，而是允许通过验证集确定一个应用相关阈值：

```text
pred = fake if fake_prob >= τ
```

其中 OOD 主实验中 `τ` 由 `ood_val.csv` 上的 `precision_0.95_recall_max` 规则决定。这样做的逻辑是：内容安全系统不能只追求最高 accuracy，而要在误报可控的前提下尽量减少 AI 视频漏检。

### 1.2 数据流与代码流

从代码运行角度看，一次训练/评估的完整数据流是：

```text
CSV row
  -> rel_path
  -> OpenCV 读取视频
  -> 均匀采样或训练随机 temporal crop
  -> frames [T,3,H,W]
  -> DataLoader batch [B,T,3,H,W]
  -> STF3Modern forward
  -> logits / aux_logits / branch_weights
  -> loss 或 fake_prob
  -> metrics.json + predictions.csv
```

对应文件如下：

| 阶段 | 代码 | 作用 |

| CSV 读取 | `src/dataset.py/read_csv_rows` | 读取 split CSV |
| 视频采样 | `src/dataset.py/read_video_frames` | OpenCV 读帧、resize、归一化到 [0,1] |
| 训练 | `src/train.py` | 训练模型，保存 `history.json`、`best.pt`、`last.pt` |

这个数据流很重要，因为它说明 notebook 不是重新实现算法，而是调用项目真实训练和评估入口。也就是说，实验结果可追溯到实际 `.py` 代码，而不是 notebook 临时逻辑。

### 1.3 为什么要拆成 S/T/F 三个分支

AI 生成视频的伪造痕迹不是单一类型。空间纹理可能显示局部不自然边缘、物体结构和语义一致性问题；时序动态可能显示帧间表征轨迹不自然、运动平滑性异常；频域统计可能显示上采样、压缩、去噪和生成模型解码过程带来的谱分布异常。三类线索的可靠性会随生成器变化而变化：

- 某些生成器空间质量很高，但时序轨迹仍不自然；
- 某些生成器帧间运动较自然，但频域压缩/重采样痕迹明显；
- 某些真实网络视频可能有压缩噪声，单独频域分支容易保守或误判；
- 某些 OOD fake 视频与训练 fake 在语义上差异很大，仅靠 spatial branch 容易漏检。

因此，STF3-New 的核心不是“堆三个模块”，而是把三类证据拆开、分别建模、再通过融合模块决定当前视频更该相信哪类证据。

### 2.1 为什么不是直接复现一个外部大模型

本项目的目标不是把 D3、ReStraV、WaveRep、SPAI 中任意一个完整搬过来，而是将它们的核心可解释线索映射到 STF3 的三分支结构中：

1. **D3 提供 temporal cue**：它强调 AI 视频在表征动态中的二阶变化异常，因此适合转化为 temporal branch 的统计特征。
2. **ReStraV 提供 representation trajectory cue**：它把逐帧 DINOv2 表征看作轨迹，因此适合补充 temporal branch 的几何特征，同时也支持使用 DINOv2 作为 spatial foundation。
3. **WaveRep 提供 wavelet/forensic cue**：它强调小波频带与取证痕迹，因此适合转化为 frequency branch 的小波统计和训练增强思想。
4. **SPAI 提供 spectral learning cue**：它更偏图像检测，但谱学习思想可以支持本项目显式统计 radial FFT bands。

这样做的好处是：方法结构清晰、算力可承受、消融实验可控，也更容易向老师解释每个分支的功能。代价是：不能声称“完整复现了这些外部方法的全部论文实验”，只能说“借鉴并实现其核心可解释线索”。

### 2.2 旧版 STF3 与新版 STF3-New 的本质区别

| 组件 | 旧版思路 | 新版 STF3-New | 变化意义 |
|---|---|---|---|
| Spatial | ResNet18 frame feature | DINOv2/CLIP/XCLIP frozen foundation encoder + adapter | 从传统 CNN 换成更强视觉基础表征 |
| Temporal | 像素级帧差 | embedding-space D3 二阶动态 + ReStraV 轨迹几何 | 从低级像素变化变成语义表征轨迹变化 |
| Frequency | FFT 图像 + SmallCNN | Haar wavelet energy + radial FFT stats | 从黑盒小 CNN 变成显式频带统计 |
| Fusion | concat + MLP | Branch-token Transformer / gated fusion | 从固定拼接变成样本级自适应证据融合 |
| Training | 单一 CE | CE + aux branch loss + fake class weighting | 更能针对分支学习和 fake 漏检问题 |

## 3. 分支功能与关键代码

### 3.1 `STF3Modern` 主结构

`src/models/stf3_modern.py` 根据 `mode` 自动启用不同分支，因此单分支、双分支、三分支消融都是同一个模型类的不同配置。

```python
if mode in MODERN_MODES:
    return STF3Modern(...)
```

`stf3_new` 启用 spatial、temporal、frequency 三个分支；`spatial_temporal_new` 只启用 S+T；`stf3_new_concat` 启用三分支但用 gated concat 作为融合对照。

更细地说，`mode` 是整个实验矩阵的开关。它不是简单改名字，而是决定模型中真实存在的分支：

| mode | Spatial | Temporal | Frequency | Fusion | 实验意义 |
|---|---:|---:|---:|---|---|
| `frequency_wave` |  |  | ✓ | linear / token fusion wrapper | 单独验证频域证据 |
| `spatial_dino` | ✓ |  |  | linear / token fusion wrapper | 单独验证 foundation spatial 表征 |
| `temporal_d3_restrav` |  | ✓ |  | linear / token fusion wrapper | 单独验证动态轨迹证据 |
| `spatial_frequency_new` | ✓ |  | ✓ | Transformer | 验证 S+F 组合 |
| `spatial_temporal_new` | ✓ | ✓ |  | Transformer | 最重要 S+T baseline |
| `stf3_new_concat` | ✓ | ✓ | ✓ | Gated concat | 验证三分支但不用 Transformer 的对照 |
| `stf3_new` | ✓ | ✓ | ✓ | Branch-token Transformer | 最终模型 |

这种 mode 设计让实验具有“同代码、同数据、同训练脚本、只改变分支组合”的公平性。尤其是 R5 与 R7 的强公平对照，正是建立在这个统一模型工厂之上。

### 3.2 Spatial branch：看空间语义、纹理和局部伪影

`FoundationVisionEncoder` 默认使用 `dinov2_vits14`，并冻结参数：

```python
feats = self.model.forward_features(x)
z = feats["x_norm_clstoken"]
return z.reshape(b, t, -1)
```

`FoundationSpatialBranch` 对每帧 embedding 做时间注意力池化：

```python
weights = torch.softmax(self.score(z).squeeze(-1), dim=1)
pooled = torch.sum(z * weights.unsqueeze(-1), dim=1)
s_token = self.adapter(pooled)
```

意义：AI 视频的伪影不一定平均分布在每帧，attention pooling 可以自动选择更有证据的帧，而不是简单平均。

Spatial branch 的输入是 `FoundationVisionEncoder` 输出的逐帧 CLS embedding。以 `dinov2_vits14` 为例，单帧输出维度为 384；经过 temporal attention pooling 后得到一个视频级空间表征，再由 adapter 投影为统一的 256 维 branch token。该分支主要承担三类信息：

1. **语义一致性**：生成视频在物体结构、场景布局、人物局部细节上可能存在不自然模式。
2. **纹理与边缘质量**：AI 生成器可能产生过平滑、过锐化或局部纹理重复。
3. **foundation representation prior**：DINOv2 这类自监督视觉模型在自然图像/视频帧上形成了强表征，AI 帧偏离该自然表征空间时可能被捕捉。

需要注意，Spatial branch 虽然使用强 foundation encoder，但默认冻结 backbone。这是一个重要工程取舍：冻结可以降低 RTX 4060 8GB 环境下的显存压力，也降低在有限训练数据上过拟合生成器特征的风险。

### 3.3 Temporal branch：看表征轨迹是否自然

D3 特征：

```python
d1 = first_order_distance(z, loss_type="l2")
d2 = second_order_delta(z, loss_type="l2")
stats = [mean(d1), std(d1), mean(d2), std(d2), mean(abs(d2)), max(abs(d2))]
```

ReStraV 特征：

```python
delta = z[:, 1:, :] - z[:, :-1, :]
d = delta.norm(dim=-1)
theta = angle(delta_t, delta_t_plus_1)
features = 7 distances + 6 angles + 8 moment statistics
```

意义：旧版时序分支用像素一阶帧差，容易受运动和压缩影响；新版在 DINOv2 表征空间看“轨迹弯曲、突变、二阶退化”，更接近生成视频的动态异常。

Temporal branch 的关键思想是：真实视频的逐帧表征变化通常来自连续物理运动、镜头变化和自然场景变化；AI 生成视频虽然单帧可能逼真，但跨帧生成机制可能导致表征轨迹不够自然。D3 和 ReStraV 分别对应两个视角：

- D3 更关注“相邻帧距离的变化是否稳定”。如果 `d1(t)=distance(z_t,z_{t+1})`，那么 `d2(t)=d1(t+1)-d1(t)` 描述的是动态变化速度本身是否突然改变。
- ReStraV 更关注“轨迹形状是否自然”。距离描述步长，转角描述方向变化，moment statistics 描述整体轨迹分布。

因此 temporal branch 不是直接判断“运动大不大”，而是判断“视频在 foundation 表征空间中的运动轨迹是否像自然视频”。

### 3.4 Frequency branch：看小波和频谱伪影

小波统计：

```python
ll, lh, hl, hh = dwt(cur)
energy = band.square().mean(dim=(1, 2, 3))
abs_mean = band.abs().mean(dim=(1, 2, 3))
```

FFT radial statistics：

```python
mag = torch.log1p(torch.abs(torch.fft.fft2(gray, norm="ortho")))
mag = torch.fft.fftshift(mag, dim=(-2, -1))
# 按频率半径分 bin，统计每个 radial band 的 mean/std
```

意义：生成、上采样、压缩和重采样会改变频域分布。小波看局部多尺度频带，FFT radial stats 看全局谱分布，两者共同构成频域 forensic cue。

Frequency branch 在本项目中尤其重要，因为它提供了与 Spatial/Temporal 不同的证据类型。空间分支和时序分支都依赖 DINOv2 表征，可能更关注语义、纹理和动态；频域分支直接从原始像素帧统计小波和频谱，因此能补充 foundation encoder 不一定显式保留的取证线索。

本项目的 frequency branch 不是完整 WaveRep 模型，也不是完整 SPAI 模型。它的定位是：

1. 用 Haar DWT 得到 LL/LH/HL/HH 多尺度频带；
2. 对每个频带统计 energy 和 absolute mean；
3. 对每个视频统计时间维 mean/std；
4. 用 radial FFT stats 补充全局频谱能量分布；
5. 最后用一个轻量 MLP adapter 将这些显式统计映射为 256 维 `f_token`。

这种设计可解释性强：如果老师问 frequency branch 到底在看什么，可以直接回答“它看多尺度小波频带能量和 radial frequency bands 的统计分布，而不是一个不可解释的 FFT 小 CNN”。

### 3.5 Fusion：Branch-token Transformer

最终 `stf3_new` 用 Transformer 融合：

```python
x = torch.stack(branch_tokens, dim=1)
cls = self.cls.expand(b, -1, -1)
x = torch.cat([cls, x], dim=1)
x = x + self.branch_embed[:, : n + 1]
enc = self.encoder(x)
logits = self.head(self.norm(enc[:, 0]))
```

同时保存 branch weights：

```python
branch_scores = torch.softmax((enc[:, 1:] * cls_out.unsqueeze(1)).sum(dim=-1), dim=1)
```

意义：concat MLP 只能粗暴拼接特征；branch-token fusion 可以让模型按样本自适应判断更信任哪个分支，尤其适合 OOD 生成器。

Branch-token Transformer 的具体输入可以写成：

```text
[CLS], [S], [T], [F]
```

其中 `[S]`、`[T]`、`[F]` 是三个 256 维 branch token。模型给不同 token 加 branch embedding，再通过两层 TransformerEncoder 进行交互。最后只取 CLS token 做分类。这种设计有两个好处：

1. **分支间交互**：模型不只是加权平均三个分支，而是允许空间、时序、频域证据之间发生条件化交互。
2. **样本级可变权重**：不同生成器、不同压缩条件、不同视频内容下，可靠证据可能不同，branch weights 可以作为解释线索。

`stf3_new_concat` 的存在也很重要。它不是最终模型，而是为了回答“提升是不是只来自三分支数量增加”。如果 gated concat 弱于 branch-token Transformer，就说明融合方式本身也有贡献。

### 3.6 模型是否免训练？权重是什么？

答案：**不是免训练模型**。它使用冻结的视觉基础模型，但训练 adapters/fusion/heads。训练设置：AdamW，lr=1e-4，weight decay=1e-4，CosineAnnealingLR，AMP，默认 batch size 1。损失包括主 CE、可选 fake 类加权 CE、aux branch loss：

```python
loss = criterion(logits, labels)
aux_loss = sum(criterion(v, labels) for v in aux.values()) / len(aux)
loss = loss + aux_loss_weight * aux_loss
```

`fake_loss_weight=1.2` 的含义：fake 样本被判错时损失乘以 1.2，目标是降低 FN。最终单模型权重是 `runs/ood_stf3_new_224_fakew12/best.pt`，由 validation AUC 选择 best checkpoint；最终部署阈值不是 0.5，而是 validation set 上校准出的 0.0545。

### 3.7 训练、评估和 predictions 文件中到底保存了什么

训练阶段保存：

| 文件 | 位置示例 | 含义 |
|---|---|---|
| `history.json` | `runs/ood_stf3_new_224_fakew12/history.json` | 每个 epoch 的 train/val loss、ACC、AUC、F1 |
| `best.pt` | `runs/ood_stf3_new_224_fakew12/best.pt` | 按 validation AUC 选择的最优 checkpoint |
| `last.pt` | `runs/.../last.pt` | 最后一轮 checkpoint |

评估阶段保存：

| 文件 | 位置示例 | 含义 |
|---|---|---|
| `metrics.json` | `outputs/ood_stf3_new_224_fakew12/metrics.json` | 默认阈值 0.5 下的 ACC/AUC/F1/Precision/Recall/confusion matrix |
| `predictions.csv` | `outputs/ood_stf3_new_224_fakew12/predictions.csv` | 每个视频的 label、fake_prob、pred、correct、generator、分支概率、branch weights |

`predictions.csv` 非常关键，因为后续阈值校准、generator-level 分析、Branch LR、ensemble 都不需要重新训练，只需要读取 val/test predictions。这样 OOD 优化才有了“先诊断，再重训”的研究过程。

### 3.8 最终模型推理流程

最终单模型在推理时的流程是：

```text
video
  -> read_video_frames(num_frames=8, image_size=224, random_sample=False)
  -> STF3Modern(mode=stf3_new)
  -> fake_prob
  -> threshold τ=0.0545
  -> real/fake
```

这里需要特别说明：训练时可能启用随机 temporal crop 和 WaveRep augmentation；但最终评估/推理时 `model.eval()`，不做随机增强，使用 deterministic sampling，保证结果可复现。

# 第二部分：实验脉络

## 4. 为什么做四类实验

| 实验 | 目的 | 主要回答 |
|---|---|---|
| Random | in-distribution sanity check | 模型和训练流程是否跑通，是否能基本检测 AI 视频 |
| OOD | cross-generator generalization | 训练生成器外的新生成器能不能识别 |
| OOD 优化 | 从错误结构出发提升最终模型 | FN 多是阈值问题、输入证据问题还是训练目标问题 |
| Comparison | 与外部方法公平对比 | STF3 的学术定位、优势和边界在哪里 |

这四类实验之间是递进关系，而不是平行堆叠。Random 先证明“模型能学”；OOD 再证明“模型能不能泛化”；OOD 优化继续追问“为什么还会失败以及怎样减少失败”；comparison 最后回答“和外部方法相比，这个方法站在哪里”。如果只写 Random，项目会像普通分类作业；如果只写 OOD，缺少训练流程 sanity check；如果只写最终最优结果，又会显得像调参。四类实验放在一起，才能形成完整科研叙事。

实验矩阵可以概括为：

```text
R1/R2/R3: 单分支，验证每类证据的独立能力
R4/R5: 双分支，验证证据组合
R6: 三分支 + gated concat，验证三分支本身
R7: 三分支 + branch-token Transformer，验证最终融合方式
follow-up: 在 R7 上进行分辨率、fake loss、阈值和推理策略优化
comparison: 与外部 D3/WaveRep/TALL 进行同协议或准同协议对照
```

## 5. Random 实验结果与意义

| ID | Method | Branches | Fusion | ACC | AUC | F1 | Precision | Recall | FN | FP |
|---|---|---|---|---|---|---|---|---|---|---|
| R1 | frequency_wave | F | - | 0.8657 | 0.9309 | 0.8440 | 0.8962 | 0.7976 | 254 | 116 |
| R5 | spatial_temporal_new | S+T | Transformer | 0.9735 | 0.9967 | 0.9705 | 0.9836 | 0.9578 | 53 | 20 |
| R7 | stf3_new | S+T+F | Branch-token Transformer | 0.9750 | 0.9963 | 0.9722 | 0.9821 | 0.9625 | 47 | 22 |

Random 的作用是证明模型基本检测能力。R5 与 R7 在 Random 上非常接近，说明同分布 split 已接近饱和；这不能否定 frequency branch，因为 Random 不足以充分检验未知生成器泛化。真正判断三分支价值要看 OOD 和强公平消融。

### 5.1 为什么 Random 只写总结果即可

Random split 中 train/val/test 共享相近的生成器家族分布，因此它更像“模型是否掌握当前数据集规律”的验证，而不是最严格的泛化评估。Random 的高分有两个作用：

1. **证明训练流程正确**：模型可以正常收敛，数据读取、标签、loss、optimizer、评估脚本没有明显错误。
2. **证明三分支不是训练不动的复杂模型**：R7 在 Random test 上 ACC=0.9750、F1=0.9722，说明 branch-token Transformer 没有造成训练困难。

但 Random 的局限也很明确：当 R5 和 R7 都接近 0.97 F1 时，分支贡献会被 ceiling effect 压缩。此时如果只看 Random，可能误以为 frequency branch 没有意义；但这是因为任务太容易，而不是因为分支没有贡献。真正的主张必须放到 OOD 和 R5_224_FAKEW12 vs R7_224_FAKEW12 的强公平对照中。

### 5.2 Random 高分是否意味着泄漏或过拟合

Random 高分需要谨慎解释，但不能直接等同于泄漏。合理解释是：Random split 中训练和测试分布接近，模型学到的生成器家族特征可以迁移到 test；同时 validation 和 test 指标接近，不表现为典型训练集高、测试集崩溃的过拟合形态。更严格的泄漏判断应检查 `rel_path`、文件名 stem、parent+stem 是否跨 split 重复；此前记忆中的检查结果未发现明显文件复用泄漏。因此，在本文档中更稳妥的表述是：

> Random split 结果证明模型在同分布场景下具有很强检测能力，但由于该设置不能充分隔离生成器分布，因此不把 Random 作为最终泛化结论的核心证据。

## 6. OOD 基础实验：统一使用 precision 阈值分支

| R1 | frequency_wave | F | - | 0.7354 | 0.7115 | 0.6742 | 0.9871 | 0.5119 | 1024 | 14 |
| R2 | spatial_dino | S | - | 0.0522 | 0.9125 | 0.9204 | 0.9801 | 0.8675 | 278 | 37 |
| R3 | temporal_d3_restrav | T | - | 0.4847 | 0.8185 | 0.8201 | 0.9719 | 0.7092 | 610 | 43 |
| R4 | spatial_frequency_new | S+F | Transformer | 0.0339 | 0.9133 | 0.9211 | 0.9811 | 0.8680 | 277 | 35 |
| R5 | spatial_temporal_new | S+T | Transformer | 0.0478 | 0.9263 | 0.9342 | 0.9756 | 0.8961 | 218 | 47 |
| R6 | stf3_new_concat | S+T+F | Gated Concat | 0.0005 | 0.9352 | 0.9425 | 0.9765 | 0.9109 | 187 | 46 |
| R7 | stf3_new | S+T+F | Branch-token Transformer | 0.3726 | 0.9202 | 0.9279 | 0.9804 | 0.8808 | 250 | 37 |

OOD 基础结论：单独 Spatial 最强，说明 foundation representation 是基础；三分支 concat/transformer 在阈值校准后明显减少 FN；但 112 输入、普通 fake loss 的基础 R7 还不是最终最优，因此需要 follow-up 优化。

### 6.1 为什么 OOD 是主实验

OOD split 的关键在于训练集、验证集和测试集的生成器分布不同。也就是说，模型不能只记住训练中见过的生成器模式，而必须利用更一般的伪造线索。这个设置更接近真实部署：现实中新的 AI 视频生成器会不断出现，检测器不可能每次都提前见过全部生成器。

因此，OOD 实验回答的问题比 Random 更难：

```text
Random: 已知或相近生成器分布下能否检测？
OOD: 未知生成器分布下能否检测？
```

这也是为什么本文最终把 OOD F1、Recall、FN/FP 和 generator-level 分析作为主证据。

### 6.2 为什么统一使用 precision_0.95_recall_max

默认 `evaluate.py` 使用 0.5 阈值，但 0.5 只是概率分类器的默认 operating point，不一定适合 OOD。尤其在内容安全任务中，系统通常希望：

1. Precision 不能太低，否则真实视频被大量误伤；
2. 在 Precision 可控的情况下，Recall 越高越好，因为 fake 漏检风险更大；
3. 阈值必须在 validation set 上确定，不能在 test set 上挑最好看的结果。

因此选择 `precision_0.95_recall_max` 作为主表 operating point。它不是为了“挑最好结果”，而是为了让所有模型在同一个应用约束下比较：

```text
在 validation set 上 Precision ≥ 0.95 的阈值中，选择 Recall 最大者。
```

这会让不同模型的决策点更公平，也更符合内容安全应用。

### 6.3 从 OOD 基础表读出的三条信息

第一，单独 `spatial_dino` 已经很强，说明 DINOv2 foundation representation 对 AI 视频检测非常重要。第二，单独 `frequency_wave` 和 `temporal_d3_restrav` 在 precision 约束下较保守，说明单一非空间线索不足以覆盖复杂 OOD。第三，三分支 `stf3_new_concat` 的 FN=187，明显少于多数单/双分支，说明三类证据组合是有价值的；但最终 `stf3_new` 基础版没有成为最优，说明结构设计之后还需要针对 OOD 的训练和阈值优化。

## 7. OOD 优化心路历程

### 7.1 从 Random 饱和转向 OOD

Random 上 R5/R7 已非常高，继续在 Random 上调参无法说明泛化。因此实验问题转为：模型能否识别 MorphStudio、Show_1、Sora、WildScrape 等未见/困难生成源。

这一转向非常重要，因为它体现了实验思维从“追求更高分”变成“追问模型真正的泛化边界”。如果一个模型只在 Random 上好，但在 OOD 上漏检大量 fake，那么它在真实内容安全系统中价值有限。因此后续 notebook 的主线不再是把所有参数都往 Random 上调，而是围绕 OOD 的错误结构展开。

### 7.2 第一个诊断：Precision 高但 Recall 低

原始 OOD 中很多模型 Precision 很高、Recall 较低，错误主要是 FN，即 fake 被漏检。内容安全场景更怕 fake 漏检，因此后续主指标转向 F1、Recall、FN/FP，而不是只看 Accuracy 或 AUC。

这个诊断决定了后续优化方向。若主要问题是 FP 过多，就应该提高阈值、加强 real 类约束或降低模型敏感度；但当前问题恰恰相反：模型太保守，很多 fake 被判为 real。因此后续优化必须围绕“如何在 Precision 可接受时提高 Recall”展开。

从风险角度看：

| 错误类型 | 含义 | 对内容安全的影响 | 本项目优化态度 |
|---|---|---|---|
| FP | real 被判成 fake | 误伤真实内容，影响用户体验 | 必须控制，不可无限增加 |
| FN | fake 被判成 real | AI 生成内容漏检，内容安全风险更大 | 当前主要要降低 |

因此本文选择 F1 和 Recall/FN 作为 OOD 叙事重点，同时保留 Precision/FP 防止模型用大量误报换取 Recall。

### 7.3 先做阈值校准

如果 AUC 高但 Recall 低，说明排序可能不错，只是默认阈值 0.5 不合适。因此先用 validation set 做 `precision_0.95_recall_max`，避免 test set cherry-picking。

这一步的科学意义是区分两个问题：

```text
问题 A：模型已经会排序，只是阈值不适合 OOD；
问题 B：模型表征本身不足，阈值怎么调都难以识别某些 fake。
```

如果阈值校准大幅改善结果，说明至少一部分瓶颈来自概率校准或 operating point；如果校准后仍有大量 FN，则说明需要改变输入证据、训练目标或模型结构。你的实验结果属于后者：阈值校准确实提升了 Recall，但 WildScrape 等困难样本仍然漏检，因此需要进入后续训练优化。

### 7.4 再做 generator-level 错误分析

错误分析发现剩余 FN 高度集中在 WildScrape。于是后续优化不再盲目追求平均提升，而是重点观察 WildScrape FN 是否下降、real_MSRVTT FP 是否可接受。

Generator-level 分析是本项目最像科研工作的地方之一。总体 F1 只能告诉我们“错了多少”，但 generator-level 能告诉我们“错在哪里”。如果 FN 均匀分布在所有 fake generator 上，说明模型整体 Recall 不足；如果 FN 集中在 WildScrape，说明存在特定困难分布，可能与真实网络视频、压缩、内容复杂度或生成器特征有关。

因此后续每个实验都不仅看总表，还要问：

1. WildScrape FN 是否下降？
2. Show_1 / Sora / MorphStudio 是否退化？
3. real_MSRVTT FP 是否可接受？
4. AUC/AP 是否也下降，还是只是阈值问题？

这使得后续负结果也能被解释，而不是简单丢弃。

### 7.5 尝试更多 epoch：不是单纯欠拟合

`R7_E6` 用 6 epoch 检验是否训练不够，但结果没有稳定超过后续最佳，说明单纯增加 epoch 不是答案。

这个实验的假设是：“也许模型还没训练够，所以 OOD Recall 不高。”如果这个假设成立，增加 epoch 后 validation/test 应该同步改善。但结果显示，更多 epoch 并没有带来决定性提升。这说明 OOD 瓶颈更可能来自训练目标、输入细节或生成器分布差异，而不是简单训练不足。

这个结果也避免了一个常见误区：看到模型不够好就盲目增加 epoch。对于 OOD，更多训练有时会让模型更贴近训练生成器，反而不一定改善未知生成器。

### 7.6 fake loss weight：针对漏检代价

由于主要错误是 fake 漏检，设置 `fake_loss_weight=1.2`，让模型在训练中更重视 fake 类错误。权重没有设得很大，是为了避免 real FP 暴涨。

`fake_loss_weight=1.2` 的设计是有目标的，而不是随便调参。普通交叉熵默认 real 和 fake 错误代价相同，但当前错误结构显示 fake 漏检更突出。因此提高 fake 类损失相当于告诉模型：把 fake 判成 real 的代价略高。

为什么是 1.2 而不是更大？因为如果 fake weight 过大，模型可能变得过于激进，把大量 real 判成 fake，导致 Precision 崩溃。1.2 是一种温和调整，用于在 Recall 和 Precision 之间移动决策边界，同时尽量不破坏 real 类稳定性。

### 7.7 image size 224：保留细粒度证据

112 快但可能丢失局部纹理、边缘和频域高频伪影。224 与 DINOv2 patch 结构更匹配，也给频域分支更多细节。

分辨率优化的动机来自三个分支共同需要的证据：

- Spatial branch 需要局部纹理和边缘；
- Temporal branch 的 DINOv2 embedding 质量受输入细节影响；
- Frequency branch 直接统计频谱，小尺寸下高频结构会被下采样削弱。

但是实验也说明，单独提高到 224 并不自动产生最优结果。224 提供了更多可观测证据，但模型是否真正利用这些证据，还取决于训练目标是否鼓励识别 fake。因此 224 与 fake loss weight 组合后才形成更明显收益。

### 7.8 224 + fake loss 的组合收益

核心 follow-up 结果如下：

| ID | Meaning | Thr | ACC | F1 | Precision | Recall | FN | FP |
|---|---|---|---|---|---|---|---|---|
| R7_E6 | R7 6 epoch：检验是否只是欠拟合 | 0.0173 | 0.9283 | 0.9357 | 0.9802 | 0.8951 | 220 | 38 |
| R7_FAKEW12 | R7 + fake_loss_weight=1.2：提高 fake 漏检代价 | 0.2457 | 0.9291 | 0.9364 | 0.9822 | 0.8947 | 221 | 34 |
| R7_224 | R7 + image_size=224：保留细粒度证据 | 0.0044 | 0.9311 | 0.9383 | 0.9813 | 0.8990 | 212 | 36 |
| R5_224_FAKEW12 | S+T 强公平对照：224 + fake_loss_weight=1.2 | 0.0089 | 0.9277 | 0.9360 | 0.9679 | 0.9061 | 197 | 63 |
| R7_224_FAKEW12 | 最终最佳单模型：S+T+F + 224 + fake_loss_weight=1.2 | 0.0545 | 0.9430 | 0.9499 | 0.9740 | 0.9271 | 153 | 52 |

最重要对照是：

```text
R5_224_FAKEW12 = S+T + 224 + fake_loss_weight=1.2
R7_224_FAKEW12 = S+T+F + 224 + fake_loss_weight=1.2
```

在相同分辨率、相同 fake loss、相同阈值校准协议下，R7 将 F1 从 0.9360 提高到 0.9499，Recall 从 0.9061 提高到 0.9271，FN 从 197 降到 153，FP 也从 63 降到 52。这是最强的三分支贡献证据。

这张表的解释应放在最终报告的核心位置。原因是它控制了最容易被质疑的变量：

| 变量 | R5_224_FAKEW12 | R7_224_FAKEW12 | 是否一致 |
|---|---:|---:|---:|
| OOD train/val/test | 相同 | 相同 | ✓ |
| image size | 224 | 224 | ✓ |
| num frames | 8 | 8 | ✓ |
| fake loss weight | 1.2 | 1.2 | ✓ |
| threshold selection | precision_0.95_recall_max | precision_0.95_recall_max | ✓ |
| 差异 | S+T | S+T+F | 频域分支与完整 STF3 |

因此，R7 的提升不能简单归因于“分辨率更高”“阈值更低”或“fake loss 更大”，因为这些在 R5/R7 中都相同。最合理的解释是：frequency branch 与三分支融合提供了 S+T 之外的互补证据。

### 7.9 最终主模型为什么不是 Branch LR 或 ensemble

Branch-level LR 和 STF3-only ensemble 的指标很好，但它们不应作为主模型本身。原因如下：

| 系统 | 优点 | 为什么不作为主模型 |
|---|---|---|
| Branch-level LR | F1=0.9530，FP=41，综合表现最好 | 使用 validation set 训练额外后处理器，不是原始 STF3 架构 |
| STF3-only ensemble | FN=148，Recall 更高 | 需要两个 checkpoint 推理，成本更高 |
| R7_224_FAKEW12 | 单模型、结构清晰、结果强 | 最适合作为论文/答辩主方法 |

因此推荐写法是：

```text
主模型：R7_224_FAKEW12
增强系统：STF3-only ensemble / Branch-level LR
负结果边界：multi-clip / WaveRep-bank / 16F
```

这样可以既展示你做了深入优化，又避免把后处理和主架构混在一起。

| R7_224_FAKEW12_5CLIP_MEAN | 5-clip mean：检验局部时间窗口 | 0.4226 | 0.9258 | 0.9337 | 0.9741 | 0.8966 | 217 | 50 |

| R7_224_FAKEW12_WAVEREP_BANK | 有效 WaveRep donor-bank：batch=1 下真正增强 | 0.1252 | 0.9283 | 0.9351 | 0.9894 | 0.8866 | 238 | 20 |
| R7_224_F16_FAKEW12 | 完整视频 16 帧：更密时序采样 | 0.0074 | 0.9339 | 0.9408 | 0.9834 | 0.9018 | 206 | 32 |
| STF3-only ensemble | R7_224_FAKEW12 + R7_224；推理增强，不是单模型 | 0.0313 | 0.9439 | 0.9508 | 0.9731 | 0.9295 | 148 | 54 |
| Branch-level LR | 用 val 学习主概率/分支概率/branch weights；后处理系统 | 0.0993 | 0.9466 | 0.9530 | 0.9794 | 0.9280 | 151 | 41 |

解释：

- STF3-only ensemble 和 Branch-level LR 证明不同 STF3 checkpoint/分支证据互补，但它们是增强系统或后处理，不是主单模型。
- Multi-clip 没有超过基准，说明简单局部时间裁剪会破坏完整视频证据。
- WaveRep-bank 让 batch=1 下增强真正生效，但结果更保守：FP 降低，FN 增加。
- 16-frame 增加训练成本但没改善 WildScrape，因此不建议作为最终主模型。

最终主模型：`R7_224_FAKEW12`。Branch LR / ensemble 可作为补充系统；WaveRep-bank、16F、multi-clip 写成负结果和边界。

### 8.1 Multi-clip 为什么失败仍然有价值

Multi-clip 的初始假设很合理：单次 8 帧采样可能错过局部伪造片段，因此从多个时间窗口取 clip，再对概率求平均或 top-k 平均，理论上可能提升 Recall。但结果没有超过基准，说明当前 STF3 不是简单依赖某个局部异常片段，而可能依赖完整视频范围内的稳定联合证据。

特别是 temporal branch 的 D3/ReStraV 统计依赖表征轨迹。如果把视频裁成多个局部 60% temporal crop，轨迹上下文可能被破坏；frequency branch 的视频级统计也可能因局部片段而偏移。因此 multi-clip 的失败不是“白跑”，它告诉我们：当前模型更适合完整视频范围内的均匀采样，而不是局部裁剪投票。

### 8.2 WaveRep-bank 为什么是必要的代码复核

WaveRep-bank 的实验价值不仅在结果，还在发现此前配置中的隐藏问题：早期主要实验使用 `batch_size=1`，而 batch 模式 WaveRep augmentation 在 batch 小于 2 时会直接返回原输入。因此虽然配置里写了 `wavelet_aug_prob=0.1`，但增强实际未发生。

这要求我们在最终解释中保持严谨：

- R7_224_FAKEW12 的 frequency branch 是有效的，因为它使用 wavelet/FFT statistics；
- 但 R7_224_FAKEW12 的提升不能归因于 WaveRep training augmentation；
- donor-bank 是为了让 batch=1 下频带替换真正发生；
- donor-bank 最终没有超过基准，因此它是负结果，而不是主贡献。

这段经历非常适合写入答辩，因为它体现了你不是只看配置表，而是回到代码层验证实验是否真的生效。

### 8.3 16-frame 为什么没有成为主模型

16-frame 实验的假设也很合理：更多帧可能让 D3/ReStraV 的轨迹统计更稳定，并减少 8 帧采样遗漏异常的概率。但实验结果显示，16F 在总体指标和 WildScrape 上没有超过 8F 基准，而且训练时间显著增加。

这说明当前问题不是“帧数越多越好”。更多帧会带来更多计算成本，也可能引入更复杂的时序噪声。对于当前 STF3 和当前 OOD split，8 帧 + 224 + fake loss weight 1.2 是更好的性价比点。

### 8.4 最终 OOD 优化心路总结

完整心路可以概括为：

```text
Random 高分
  -> 发现 Random 已接近饱和
  -> 转向 OOD
  -> 发现 OOD Precision 高但 Recall 低
  -> 用 val 阈值校准判断是否只是阈值问题
  -> 发现仍有 WildScrape 漏检
  -> 尝试更多 epoch，收益有限
  -> 针对 FN 提高 fake loss weight
  -> 针对细粒度证据提高 image size
  -> 组合得到 R7_224_FAKEW12
  -> 用 R5_224_FAKEW12 做强公平消融
  -> 尝试后处理和增强系统
  -> 用负结果确定优化边界
```

这条链条比单纯“我跑了很多实验”更有说服力，因为每个实验都回答了上一个实验留下的问题。

# 第三部分：Comparison 对比实验

## 9. 为什么选择这些外部方法

| 方法 | 选择原因 | 与 STF3 的关系 |
|---|---|---|
| D3 | training-free second-order dynamics，代表纯时序退化线索 | 对应 temporal branch 的 D3 灵感 |
| WaveRep | 官方预训练取证/频域相关检测器 | 对应 frequency branch 的小波/频谱线索 |
| TALL-SWIN | 强外部可训练时空 baseline，Swin + thumbnail layout | 检验强时空 backbone 与可解释 STF3 的差距 |

巧思是：三种外部方法分别对应 temporal cue、frequency/forensic cue、strong spatiotemporal cue，而不是随意堆 baseline。

更具体地说，comparison 的设计遵循三个层次：

1. **机制相近但更单一的方法**：D3 只看 second-order dynamics，因此可以检验“仅靠时序退化线索是否足够”。
2. **外部预训练取证模型**：WaveRep 使用官方预训练权重，代表更强的外部 forensic detector，但没有在本项目 OOD train 上重训。
3. **强可训练时空模型**：TALL-SWIN 在同一 OOD train/val/test 协议下训练，是更严格的外部强 baseline。

因此，comparison 的目标不是让 STF3 在所有列都第一，而是明确：

- STF3 是否比单一 D3 动态线索更完整；
- STF3 是否比外部预训练 WaveRep 更少漏检；
- STF3 与强 TALL-SWIN 相比，到底优势和弱点分别在哪里。

### 9.1 公平性说明

外部对比最容易被质疑的是数据协议是否一致。本项目中需要这样解释：

| 方法 | 是否使用 OOD train | 阈值如何选择 | 公平性定位 |
|---|---:|---|---|
| STF3 | 使用 | OOD val | 主方法 |
| D3 | 不使用，training-free | OOD val | 外部免训练 temporal baseline |
| WaveRep | 不使用本项目 train，使用官方预训练 | OOD val | 外部预训练 forensic baseline |
| TALL-SWIN | 使用 OOD train | OOD val | 强外部可训练 baseline |

这意味着 D3/WaveRep/TALL 的性质不同，不能简单说“谁绝对公平”。更严谨的写法是：在同一个 OOD val/test evaluation protocol 下比较它们的 operating point，同时清楚标注它们是否使用了本项目训练集或外部预训练权重。

## 10. 主对比表（precision 阈值分支）

| Type | Method | Test N | AUC | F1 | Precision | Recall | FN | FP |
|---|---|---|---|---|---|---|---|---|
| Proposed | STF3-New R7_224_FAKEW12 | 3598 | 0.9848 | 0.9499 | 0.9740 | 0.9271 | 153 | 52 |
| Training-free | D3 XCLIP-16 L2 | 3585 | 0.9513 | 0.7662 | 0.9806 | 0.6288 | 774 | 26 |
| Pretrained | WaveRep DINOv2 G4 | 3585 | 0.9573 | 0.9207 | 0.9758 | 0.8715 | 268 | 45 |
| Trainable | TALL-SWIN 3ep | 3585 | 0.9898 | 0.9549 | 0.9882 | 0.9237 | 159 | 23 |

严谨解读：

1. STF3 明显优于 D3 和 WaveRep 的 Recall/FN 控制，说明单一动态线索或外部预训练频域检测器在当前 OOD 上仍偏保守。
2. TALL-SWIN 是非常强的外部可训练 baseline。按当前 CSV 的 precision 分支，TALL 的 AUC、F1、Precision 和 FP 控制很强。
3. STF3 的优势不是“所有指标全面碾压”，而是可解释 S/T/F 三证据融合，并在 Recall/FN 控制上与 TALL 接近甚至略优（FN 153 vs 159）。

答辩中建议表述：STF3 是一个可解释、模块化、跨证据融合框架；它显著强于 D3/WaveRep，并与强 TALL baseline 竞争，但不夸大为全面超过所有外部模型。

### 10.1 对每个外部方法的答辩式解释

**D3**：D3 的 AUC=0.9513，说明 second-order dynamics 本身确实有信息；但在 precision 约束下 Recall=0.6288，FN=774，说明单一动态退化线索在当前 OOD split 上过于保守。STF3 的优势在于把 D3-like temporal cue 与 spatial/frequency cue 结合，从而显著减少漏检。

**WaveRep**：WaveRep 的 Precision=0.9758，说明其外部预训练取证线索可靠，但 Recall=0.8715、FN=268，仍比 STF3 漏检更多 fake。这个对比支持 frequency cue 有价值，但也说明单独依赖外部频域/取证模型不如 S/T/F 融合稳定。

**TALL-SWIN**：TALL 是最强外部 baseline。它的 F1=0.9549，高于 STF3 的 0.9499，说明强 Swin backbone + thumbnail layout 在当前数据上非常有效。STF3 的优势不是全面压过 TALL，而是在 FN 控制上接近甚至略好，同时模型结构更可解释：可以分别分析 spatial、temporal、frequency evidence。

### 10.2 最推荐的 comparison 结论写法

推荐写成：

> 在统一 OOD validation-threshold evaluation protocol 下，STF3-New 显著优于 D3 和 WaveRep 的 fake recall 与 FN 控制，说明单一时序退化或外部频域取证线索不足以覆盖当前跨生成器场景。与强 TALL-SWIN baseline 相比，STF3-New 的 F1 略低，但 FN 略少，并提供更明确的 S/T/F 证据分解。因此，STF3-New 的主要贡献不是声称全面超过所有强外部模型，而是提供一个可解释、模块化且具有竞争力的三证据融合框架。

> STF3 在 AEGIS 上具有较强的 AI 视频召回能力，但完整二分类性能受到真实视频误报和跨数据集分布偏移限制。后续优化重点应放在真实视频 hard negative 扩充、阈值迁移校准和跨数据集 domain shift 适应上。

## 10.4 RealRobust-UCF101：真实视频增强的负结果

AEGIS 外部测试暴露 STF3 的核心问题是：AI 视频召回较强，但真实视频误报较多。为验证“增加真实视频是否能降低 FP”，项目进一步进行了 RealRobust-UCF101 实验。

实验设置：

```text
Train = GenVideo OOD train + 2000 个 UCF-101 real videos
Val   = GenVideo OOD val
Test  = AEGIS hard test set
```

该实验不使用 AEGIS 参与训练，AEGIS 仍作为外部测试集。实验目标不是继续提高 AI 召回，而是降低真实视频误报、提升 Precision 与 Specificity。

### 10.4.1 实验结果

| 模型 | ACC | AUC | F1 | Precision | Recall | Specificity | TN | FP | FN | TP |
|---|---:|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| Original STF3-New R7_224_FAKEW12 | 0.6858 | 0.7184 | 0.7523 | 0.6209 | 0.9541 | 0.4174 | 91 | 127 | 10 | 208 |
| RealRobust-UCF101 | 0.6514 | 0.7095 | 0.7185 | 0.6025 | 0.8899 | 0.4128 | 90 | 128 | 24 | 194 |

### 10.4.2 结论

RealRobust-UCF101 没有达到预期：

- FP 从 127 增加到 128，真实视频误报没有降低；
- Specificity 从 0.4174 降至 0.4128，真实视频识别能力没有提升；
- FN 从 10 增加到 24，AI 视频漏检增加；
- Recall 从 0.9541 降至 0.8899；
- ACC、AUC、F1、Precision 均低于原始 STF3。

因此，该实验应作为 **负结果** 记录，不应替代主模型 `R7_224_FAKEW12`。

### 10.4.3 为什么这个负结果有价值

这个负结果说明：**降低外部真实视频误报不能简单依赖增加普通 real 数据**。

UCF-101 虽然是真实视频数据集，但它主要面向动作识别任务，视频来源、拍摄风格、压缩方式和 AEGIS 的 `real/youtube`、`real/dvf` 并不完全一致。因此，加入 UCF-101 并没有让模型学会区分 AEGIS 中容易被误报的真实视频。

更严谨的结论是：

> AEGIS 误报问题更可能来自目标域真实视频分布偏移，而不是训练集中 real 数量简单不足。后续应使用更接近目标域的 real hard negatives，或进行阈值迁移与 domain calibration，而不是盲目堆叠普通真实视频。

### 10.4.4 可用于答辩的说法

> 我尝试加入 2000 个 UCF-101 真实视频来降低 AEGIS 上的真实视频误报，但结果显示 FP 没有下降，反而 Recall、F1 和 ACC 下降。这说明问题不是简单的 real 数量不足，而是 hard negative 分布不匹配。这个负结果帮助我明确下一步优化方向：需要更接近 AEGIS real/youtube、real/dvf 的真实视频 hard negatives，而不是盲目加入通用动作识别视频。

# 第四部分：最终结论与边界

## 11. 推荐主线故事

1. AI 生成视频检测不能只看 Random split；真实部署更需要 OOD。
2. STF3 把证据分成 Spatial、Temporal、Frequency，分别对应语义纹理、动态轨迹和频谱伪影。
3. Branch-token Transformer 让三类证据按样本自适应融合。
4. Random 证明训练流程与基础能力；OOD 暴露 FN/Recall 瓶颈。
5. 阈值校准和 generator 分析定位问题；224 + fake_loss_weight=1.2 形成最终最佳单模型。
6. 强公平消融证明三分支 STF3 优于 S+T 双分支。
7. 外部对比证明 STF3 的竞争力与边界；负结果证明不是盲目堆模块。
8. AEGIS 外部 hard test set 进一步暴露跨数据集 domain shift：STF3 的 AI 视频召回较强，但真实视频误报偏高，因此后续优化应围绕真实视频 hard negatives 和跨数据集校准展开。
9. RealRobust-UCF101 作为后续优化负结果，证明简单加入普通真实视频不能直接解决 AEGIS 误报，需要更接近目标域的 hard negatives。

如果需要答辩口头讲述，可以压缩成下面 90 秒版本：

> 我最初先在 Random split 上验证模型能否学到 AI 视频检测任务。结果 R5 和 R7 都接近饱和，所以我意识到 Random 不能充分证明泛化，于是转向更难的 OOD split。OOD 里我发现模型普遍 Precision 高但 Recall 偏低，主要问题是 fake 漏检。因此我没有直接盲目重训，而是先做 validation threshold calibration，确认一部分问题来自阈值，但 WildScrape 仍是主要失败来源。接着我围绕两个假设优化：第一，提高输入到 224 保留更多细粒度空间和频域证据；第二，用 fake_loss_weight=1.2 提高模型对 fake 漏检的惩罚。二者组合得到当前最佳单模型 R7_224_FAKEW12。为了证明不是单纯参数带来的提升，我又做了 R5_224_FAKEW12 与 R7_224_FAKEW12 的强公平消融，在相同分辨率、相同 fake loss 和相同阈值协议下，完整三分支 STF3 比 S+T 双分支少 44 个 FN。最后，我用 D3、WaveRep 和 TALL 做外部对比，说明 STF3 在可解释三分支融合和漏检控制上有优势，同时也承认 TALL 在部分指标上更强。进一步地，我加入 ACM MM 2025 的 AEGIS hard test set 作为外部泛化诊断，发现 STF3 对 AI 视频召回很高，但真实视频误报较多。随后我尝试加入 UCF-101 真实视频做 RealRobust 训练，但结果反而下降，说明问题不是简单 real 数量不足，而是 hard negative 分布不匹配。

## 12. 可直接用于报告的结论段

本项目将 AI 生成视频检测建模为跨生成器泛化问题，而非仅在随机划分上追求高准确率。STF3-New 将视频中的空间语义、时序动态和频域伪影分别编码为 branch token，并通过 Branch-token Transformer 进行样本级自适应融合。实验表明，Random split 上模型具备稳定的基础检测能力；在更具挑战的 OOD split 上，原始结果暴露出高 Precision 但 Recall 不足的问题。通过验证集阈值校准、generator-level 错误分析以及针对 FN 的训练优化，最终得到 `R7_224_FAKEW12` 单模型。与相同 224 输入、相同 fake loss weight 和相同阈值校准协议下的 spatial-temporal baseline 相比，完整 STF3 将 OOD F1 从 0.9360 提升到 0.9499，将 Recall 从 0.9061 提升到 0.9271，并将 FN 从 197 降至 153。进一步外部对比显示，STF3 显著优于 D3 和 WaveRep 的漏检控制，并与 TALL-SWIN 强 baseline 具有竞争力。新增 AEGIS hard test set 外部实验进一步说明，STF3 在跨数据集场景下仍保持较高 AI 视频召回率（Recall=0.9541），但真实视频误报明显偏高（FP=127），暴露出 domain shift 与真实视频分布覆盖不足的问题。RealRobust-UCF101 后续实验未能改善 AEGIS 误报，反而使 ACC、F1、Precision 和 Recall 下降，说明简单加入普通真实视频并不能替代目标域 hard negative mining。负结果实验和 AEGIS 诊断共同表明，当前收益主要来自三分支互补证据、224 细节保留和召回导向训练目标的组合，而后续优化应重点关注真实视频 hard negatives、阈值迁移和跨数据集校准。

## 13. 结论边界

可以支持：

- STF3-New 是需要训练的三分支模型，foundation encoder 默认冻结。
- `R7_224_FAKEW12` 是当前最佳 STF3 单模型。
- 强公平消融支持 Frequency branch + STF3 fusion 对 OOD 有贡献。
- Branch LR 和 ensemble 说明内部证据互补，但属于后处理/增强系统。
- WaveRep-bank、16-frame、multi-clip 是有价值的负结果。
- AEGIS 外部测试支持“STF3 具有高召回初筛能力”，但同时暴露真实视频误报问题。
- RealRobust-UCF101 是有价值的负结果，说明普通 UCF-101 real 不能直接解决 AEGIS 误报。

不能过度声称：

- 不能说 STF3 在所有指标上全面超过 TALL。
- 不能说 WaveRep augmentation 贡献了 R7_224_FAKEW12；早期 batch=1 下它实际未生效。
- 不能说 224 或 fake loss 在所有数据集必然有效。
- 不能把 Branch-level LR 写成主模型架构。
- 不能说 STF3 在 AEGIS 上整体表现优异；更准确的说法是 AI 视频召回强，但 real 视频误报多。
- 不能把 RealRobust-UCF101 写成优化成功；它是负结果，不替代主模型。

## 13.1 可能被老师问到的问题与建议回答

| 问题 | 建议回答 |
|---|---|
| 你的模型是不是免训练？ | 不是。DINOv2 foundation encoder 默认冻结，但 adapters、fusion、分类头和辅助头需要训练。D3/WaveRep/ReStraV 是方法灵感，不代表 STF3-New 免训练。 |
| 为什么 Random 上 R5/R7 差距很小，还说三分支有意义？ | Random 已接近饱和，不能充分区分泛化。强证据来自 OOD，尤其是 R5_224_FAKEW12 vs R7_224_FAKEW12 的公平对照。 |
| 为什么选择 precision_0.95_recall_max？ | AI 视频检测是内容安全任务，需要在误报可控时尽量减少 fake 漏检。阈值只在 val 上选，再固定到 test，避免 test cherry-picking。 |
| WaveRep 增强是不是贡献了最终结果？ | 不是。早期 batch=1 时 WaveRep batch augmentation 实际未生效；后续 donor-bank 让它生效，但未超过基准。最终提升主要来自三分支结构、224 和 fake loss。 |
| Branch LR 结果最好，为什么不作为最终模型？ | 它是 validation-fitted 后处理器，不是主网络结构。可以作为增强系统报告，但主模型应保持为单 checkpoint 的 R7_224_FAKEW12。 |
| TALL 比你强吗？ | TALL 在 AUC/F1/Precision/FP 上很强，不能说 STF3 全面超过它。但 STF3 在 FN/Recall 控制上竞争力强，并且提供更清晰的 S/T/F 可解释分支。 |
| 你的 frequency branch 到底有什么用？ | 最强证据是相同 224、fake loss、阈值协议下，加入 frequency 的 R7 比 S+T 的 R5 少 44 个 FN；说明 frequency evidence 在 OOD 中提供互补信息。 |
| AEGIS 上 STF3 为什么 ACC 不高？ | AEGIS 是外部 hard set，真实视频来源与 GenVideo 差异更大。STF3 的 fake Recall=0.9541、FN=10，说明 AI 检出很敏感；但 FP=127，说明真实视频误报较多，后续需要真实视频 hard negatives 和跨数据集校准。 |
| 加入 UCF-101 真实视频后为什么没提升？ | UCF-101 是动作识别数据集，虽然是真实视频，但不一定接近 AEGIS 的 real/youtube、real/dvf 分布。实验中 FP 没有下降，FN 反而增加，说明需要目标域 hard negatives，而不是简单增加普通 real 数量。 |
